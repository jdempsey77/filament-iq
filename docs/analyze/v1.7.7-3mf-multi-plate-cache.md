### 0.1 What does `_try_cache_3mf` do today?

File: `apps/filament_iq/ams_print_usage_sync.py`, lines 1613–1722.

**File-discovery strategy (current, as shipped):**

The current production code already includes the multi-plate glob logic introduced as part of the v1.7.7 work-in-progress (the method body at lines 1679–1706 shows the glob). This means the cache file is not simply read as a single path. The discovery strategy is:

1. Read the `gcode_file_downloaded` sensor entity to get the filename (e.g., `1-TaskName.gcode`). Line 1624–1629.
2. Strip `.gcode`, extract the stem (e.g., `1-TaskName`). Line 1634.
3. Parse the stem to separate `plate_id` (`1`) from `task_name` (`TaskName`) using the first dash. Lines 1636–1640.
4. Cross-validate: read the `task_name` entity and confirm it matches the portion extracted from the gcode filename. Lines 1642–1655.
5. Construct the primary single-plate path: `{cache_path}/{stem}.slice_info.config` (e.g., `1-TaskName.slice_info.config`). Lines 1657–1658.
6. Verify the file exists and its mtime is >= `print_start_time - 60s`. Lines 1660–1672.
7. Parse the primary file via `parse_slice_info_file`. Line 1674.
8. **Glob for sibling plates:** `{cache_path}/*{task_name}*.slice_info.config`. Lines 1681–1687. Filter candidates to those also passing the mtime guard and not equal to the primary path.
9. Merge sibling filaments into `merged` list (see 0.1 merge logic below). Lines 1688–1699.
10. Set `self._threemf_data = merged`. Line 1708.

**Current merge / aggregation logic (`merged` at line 1708):**

`merged` starts as a copy of the primary plate's filament list (line 1688). For each `extra_filament` from a sibling plate:
- Look up `tray_info_idx` in the existing `merged` list.
- If not found: append the entry (new slot used by this plate).
- If found and `ef["used_g"] > existing["used_g"]`: replace the existing entry with the higher-usage entry (line 1698).
- If found and `ef["used_g"] <= existing["used_g"]`: silently discard.

This is a **max-wins dedup** by `tray_info_idx`.

**Current dedup / collision behavior on duplicate `tray_info_idx`:**

As described above: the entry with the higher `used_g` wins. No logging of per-slot dedup decisions.

**Current log emissions:**

- `3MF_CACHE_MISS reason=*` — various miss reasons (lines 1628, 1632, 1638, 1645, 1651, 1661, 1668).
- `3MF_CACHE_MULTI_PLATE plate_count=N paths=[...]` — emitted only when `additional_paths` is non-empty (lines 1700–1706).
- `3MF_CACHE_HIT path=... filaments=N total_g=...` — emitted on success (lines 1712–1716).
- `3MF_CACHE_ERROR reason=...` — emitted on exception (line 1721).

**Note:** The `3MF_CACHE_MULTI_PLATE` log is emitted before the `3MF_CACHE_HIT` log, meaning multi-plate merges are visible in sequence. Neither the primary plate name nor sibling names are logged beyond the basename list in the multi-plate log.

### 0.3 The two rehydration sites — what's the difference?

**Rehydration Site 1 — `_on_print_finish` (line 1159):**

Call path: `_on_print_status_change` → new status is terminal (`finish`, `failed`, etc.) → `_lifecycle_phase2` → `_on_print_finish(new_status)` (line 853–854). Inside `_on_print_finish`, after the dedup guard passes, the code checks `self.threemf_enabled and self._threemf_data is None` at line 1156. This is a **late disk recover at print finish** — the app was mid-print but lost `_threemf_data` from memory. It attempts `_load_active_print`, and if that also fails, falls through to `_try_cache_3mf` at line 1189. After this block, execution continues directly to `_do_finish()` at line 1197.

This is NOT a full rehydration. `_rehydrate_print_state()` is not called. This path only restores `_threemf_data` (and `_threemf_filename`, `_trays_used`, `_spool_id_snapshot`, `_print_start_time`) at the last moment before consumption calculation. It is triggered during normal operation when AppDaemon restarted after print start but before print finish.

**Rehydration Site 2 — `_rehydrate_print_state` (line 1523):**

Call paths:
1. `initialize()` → AppDaemon app startup → `_rehydrate_print_state()` (line 1426) is called when printer is found mid-print (status in `running/printing/pause/paused`).
2. `_on_ha_start()` (line 1557) → called on `homeassistant_start` event → `_rehydrate_print_state()`.

This is the **early rehydration path** — HA or AppDaemon just (re)started while a print is in progress. It rebuilds the full in-memory state: `_print_active`, `_start_snapshot`, `_trays_used`, `_spool_id_snapshot`, `_job_key`, `_print_start_time`, and now `_threemf_data`/`_threemf_filename`/`_threemf_from_disk_restore`.

After the disk restore attempt at line 1523, there is an immediate fallthrough to `_try_cache_3mf` at lines 1551–1555 if `_threemf_data is None`. This means: if `active_print.json` is absent or stale, the cache is tried immediately on rehydration.

**Are they duplicated logic?**

Structurally they are nearly identical (lines 1159–1186 mirror lines 1523–1555). Both:
- Call `_load_active_print(_job_key)`
- If restored, set `_threemf_data`, `_threemf_from_disk_restore`, `_threemf_filename`, `_trays_used`, `_spool_id_snapshot`, `_print_start_time`
- If `_threemf_data is None` after that, try `_try_cache_3mf()`

The difference is context: Site 1 is a last-ditch attempt at finish time; Site 2 is an early state restoration at startup time. Site 2 also resets the full lifecycle state (builds start snapshot, seeds active trays, sets `_rehydrated = True`, etc.) before attempting the disk restore, which Site 1 does not do. They are NOT functionally equivalent and merging them would require care.

### 0.5 The "double-write guard" at line 1730

**Current guard (lines 1730–1741):**

```python
# Double-write guard: if _threemf_data was set externally, skip everything.
# Exception: if data came from a single-plate disk restore, allow cache
# re-check to merge any additional plates found since the restore.
if self._threemf_data is not None:
    if self._threemf_from_disk_restore:
        self.log("3MF_CACHE_DISK_RESTORE_RECHECK — running cache check", level="INFO")
        if self._try_cache_3mf():
            return
        # Cache miss: fall through to FTPS
    else:
        self.log("3MF_CACHE_ALREADY_SET — skipping FTPS", level="INFO")
        return
```

**What the original guard protected against:**

The guard was introduced in v1.7.0 to prevent FTPS from running a redundant fetch after the local cache already succeeded. Without it, FTPS would run 10s after print start, complete successfully, and overwrite the cache result — wasting network and thread resources. It also prevented a race where a slow FTPS response for a *different* print could overwrite a fresh cache hit for the *current* print (though the job_key stale check at line 1930 provides a secondary guard for that).

**What narrowing the guard (allowing disk-restore recheck) means for those invariants:**

The narrowing at line 1734 allows `_try_cache_3mf()` to run even when `_threemf_data is not None`, but only when `_threemf_from_disk_restore = True`. This is safe with respect to the original anti-overwrite intent because:

1. The disk-restore flag is only set at rehydration time (sites 2 and 3). A print that was freshly started (not rehydrated) will have `_threemf_from_disk_restore = False`, and the `ALREADY_SET` branch fires as before.
2. `_try_cache_3mf()` explicitly sets `_threemf_from_disk_restore = False` on any cache hit (line 1710), so a second call to `_fetch_3mf_background` after a successful recheck will take the `ALREADY_SET` path — no infinite recheck loop.
3. If the recheck is a cache miss (line 1737–1738), execution falls through to FTPS. The FTPS write at site 6 (line 1956) does NOT reset `_threemf_from_disk_restore`, which is the latent inconsistency noted in 0.2.

**What remains unprotected:**

The guard does not protect against the case where a disk-restore recheck runs, finds a cache miss, falls through to FTPS, and then FTPS also fails. In that case `_threemf_data` may retain the disk-restored value (it was not cleared by the recheck path). This is arguably correct behavior — disk-restored data is better than nothing — but it is not documented.

## Section 1: Skeptic (R1) Review

### Q1: Is "max used_g wins" the right dedup rule?

**The scenario:** A print job slices multiple plates. Plate 1 uses 5g of slot 0 (a small test piece on the same plate as other colors). Plate 2 uses 18.5g of slot 0 (a larger model). The slicer writes one `slice_info.config` per plate. Each file accurately reports usage for *that plate only*.

**The max-wins rule:** The merged entry for `tray_info_idx = "0"` becomes 18.5g.

**Argument for max-wins:**

In the common multi-plate print pattern where all plates are printed in sequence as a single job, the total consumption for slot 0 would be 5g + 18.5g = 23.5g. Max-wins gives 18.5g, which is wrong if both plates printed.

However, `tray_info_idx` in the Bambu slicer represents the *slot assignment* for the filament type, not a unique per-plate usage record. The slicer outputs per-plate usage for each slot that participates in that plate. So if a 6-plate job uses slot 0 on every plate, there are 6 records with the same `tray_info_idx`, and summing them gives total consumption across all plates. This is the physically correct semantic.

**The pre-existing `merged` logic at line 1688:** It started from the *primary plate* filament list. For additional plates it does max-wins, not sum. This means even the "before" state (before v1.7.7) was broken for the accumulation case — a primary plate with 5g and a sibling with 18.5g would produce 18.5g (max), not 23.5g (sum).

**Argument against max-wins:**

Max-wins is defensible as a heuristic *only* if it is designed to handle a specific known failure mode: the slicer emitting duplicate entries for the same plate across multiple cache files (e.g., partial writes, retries, or ha-bambulab writing the same slice_info.config twice). In that case, max-wins prevents double-counting.

But it does NOT handle the primary case: a genuine multi-plate print where the same slot is used across multiple plates. In that case, summing is correct and max-wins systematically under-reports consumption.

**The test encodes wrong behavior.** `test_try_cache_3mf_multi_plate_dedup_keeps_max_used_g` asserts that 18.5g wins over 5g. This will under-report consumption for any multi-plate print where a slot is used on multiple plates. For a 6-plate print all using slot 0, the per-plate files might be: 5g, 18.5g, 12g, 8g, 22g, 15g. Max-wins would give 22g; sum would give 80.5g. The spool would be visibly not depleted at the expected time.

**The true question for Jerry:** Does the Bambu slicer's `slice_info.config` represent cumulative usage for the whole job split across per-plate files, or per-plate usage independently? If the slicer already sums all plates into a *single* file and the per-plate files are redundant slices of it, then max-wins (really: take any one file, they're equivalent) is correct. If the per-plate files represent independent contributions that must be summed, max-wins is wrong.

The existing test does not prove which model is correct — it only proves that the implementation follows max-wins.

**Alternatives:**
- **Sum:** 23.5g — correct if each plate file reflects independent consumption for that plate.
- **Last-plate-wins:** 18.5g or 5g depending on glob order — non-deterministic, wrong.
- **Flag-as-ambiguous:** Return a `confidence=low` entry or log a warning when a `tray_info_idx` collision is detected — conservative but loses data.
- **Max-wins:** 18.5g — correct only in the dedup-artifact scenario, wrong in the multi-plate accumulation scenario.

**Recommendation:** This is a spec-level question that must be answered by examining what the Bambu slicer actually writes in practice. Until then, the test at line 3645 must be treated as unconfirmed. See Open Question 1.

### Q2: Does `_threemf_from_disk_restore=True` regress v1.7.4 set-once invariants?

**Walking the scenario:**

- **T0:** Print starts. `_on_print_start()` fires. `_print_start_time` set. `_job_key` set. `_fetch_3mf_background` scheduled in 10s. Cache hit at T10: `_threemf_data` set, `_threemf_from_disk_restore = False`. `_persist_active_print()` writes `active_print.json`.

- **T1:** HA blip (integration reload / `homeassistant_start` event). `_on_ha_start` fires → `_rehydrate_print_state()`. Print status is still `running`. App restores state from `active_print.json`. At line 1523: `_threemf_data = restored["threemf_data"]` (the merged data from T0), `_threemf_from_disk_restore = True`.

- **T2:** `_rehydrate_print_state()` is done. If `_threemf_data` was restored from disk (non-None), lines 1551–1555 do NOT call `_try_cache_3mf()` (the guard is `if self._threemf_data is None`). So the disk-restore flag is set but no immediate recheck happens at startup.

- **T3:** `_fetch_3mf_background` is NOT rescheduled during rehydration. The only trigger for `_fetch_3mf_background` is `_on_print_start()` at line 822, which is guarded by the set-once check. On rehydration via `_rehydrate_print_state`, `_on_print_start()` is NOT called directly — the startup path only restores state. So: `_fetch_3mf_background` does NOT run again after rehydration (unless a print status change re-triggers it, which requires a `none/idle → running` transition, not a blip).

  **This means the disk-restore recheck path at lines 1733–1741 is only triggered if `_fetch_3mf_background` is called.** It is called via `run_in(self._fetch_3mf_background, 10)` at line 822, which fires only on `_on_print_start()`. After a blip-and-rehydrate, if the print status sensor blips back through a non-running state to running, `_on_print_start()` would be called again — but the set-once guard on `_print_start_time` and `_job_key` would no-op it. However, `_fetch_3mf_background` is scheduled via `run_in` inside `_on_print_start()` at line 822, BEFORE the set-once guard check. Let me verify this...

  Examining line 820–824:
  ```python
  if self.threemf_enabled:
      self.run_in(self._fetch_3mf_background, 10)
  if self._lifecycle_phase1:
      self._on_print_start()
  ```
  The `run_in` for `_fetch_3mf_background` is NOT inside `_on_print_start()`; it is in `_on_print_status_change()` inside the `new in ("running", "printing")` branch. The set-once guard in `_on_print_start()` covers `_print_start_time` and `_job_key`, but `_fetch_3mf_background` is scheduled before `_on_print_start()` is called. If `_on_print_status_change` fires again on a blip (e.g., `idle → running` after a sensor dropout), `_fetch_3mf_background` WILL be rescheduled (line 822), and if `_threemf_from_disk_restore = True` at that point, the recheck runs.

- **T3 (revised):** If a blip causes `idle → running` transition after rehydration, `_fetch_3mf_background` is scheduled. At T3+10s, it fires. `_threemf_data is not None` and `_threemf_from_disk_restore = True`, so the recheck branch runs (line 1734). `_try_cache_3mf()` is called, which may return DIFFERENT data (new sibling plate files have appeared since the original T0 cache hit).

- **T4:** If the recheck hits a new/different 3MF, `_threemf_data` is updated. `_threemf_from_disk_restore = False`. Print finishes. `_do_finish()` runs with the new 3MF.

**Could this cause double-counting, under-counting, or slot mis-attribution?**

Double-counting: No — `_do_finish()` calls `match_filaments_to_slots()` at line 1267 once at print finish time using whatever `self._threemf_data` holds at that moment. There is no intermediate "commit" of consumption between T3 and T4. The consumption decision is made exactly once at finish time.

Under-counting: Yes, potentially. If the T0 cache hit had a complete multi-plate merge (say, 5 plates) and the T3 recheck runs but the ha-bambulab cache directory has been partially cleared (see Q5), the recheck might return fewer filaments or a stale single-plate result. `_threemf_data` would be overwritten with inferior data.

Slot mis-attribution: Possible if max-wins dedup is wrong (see Q1) and the recheck returns a different plate's config file as the primary, causing different filament entries to "win" the dedup.

**Does the v1.7.4 set-once invariant regress?**

The v1.7.4 invariant covers `_print_start_time` and `_job_key`. Neither is touched by the recheck path. The recheck only affects `_threemf_data`, `_threemf_filename`, and `_threemf_from_disk_restore`. So the v1.7.4 invariant is not formally regressed.

However, a new implicit invariant is introduced: `_threemf_data` is now mutable after rehydration, where before it was effectively set-once (FTPS or cache, whichever fired first, won). The recheck creates a second mutation window. This must be documented.

### Q3: Are there really two rehydration sites that need the flag?

Yes. Both sites 2 (line 1159, in `_on_print_finish`) and 3 (line 1523, in `_rehydrate_print_state`) set `_threemf_from_disk_restore = True`. Both are required.

### Q5: Cache directory stability during the rehydration window

The cache directory at `{bambulab_cache_path}` (configured as `/homeassistant/www/media/ha-bambulab/{serial}/prints/cache/` per CHANGELOG v1.7.1) is written by the ha-bambulab HA integration. The following events can invalidate it:

1. **ha-bambulab restart:** The integration may re-sync cache files on restart. It may write new files (adding sibling plates) or clear stale files (removing files from old prints). If the recheck runs during a ha-bambulab restart, glob results could be inconsistent — some plates present, some not.

2. **Print finishing on the printer side:** When the print completes, the printer may update the gcode_file sensor. If the task_name or gcode_file sensor changes between the disk restore and the recheck, `_try_cache_3mf()` will fail the task_mismatch check at line 1649 and return False (cache miss), falling through to FTPS. This is actually safe behavior — the mismatch guard prevents stale reads.

3. **Weekly Sunday 02:00 cache-clearing cron:** If this runs during a long print (plausible for multi-day prints), the cache directory is cleared. The disk-restore recheck would find no files and return False, falling through to FTPS. Safe, but a multi-day print would lose the merged multi-plate data.

4. **The ha-bambulab integration writing plate 2+ files with a delay:** The most relevant case. After a multi-plate print starts, ha-bambulab may write plate 1's config immediately and plate 2–N configs with some delay. If the original cache hit (T0) only saw plate 1, `active_print.json` has single-plate data, `_threemf_from_disk_restore` is False, and the recheck path is never triggered. The recheck is only triggered when the *disk restore* gets single-plate data. If the original cache hit (not disk restore) was single-plate, the recheck does not run.

This is a meaningful gap: the disk-restore recheck helps only when HA restarted after the original single-plate cache hit. If HA does NOT restart and the original cache hit was single-plate, the multi-plate merge never happens.

### Section 1 Conclusion

**Objections that must be resolved before merge:**

1. **[BLOCKER] Max-wins dedup semantic is unconfirmed.** The test at line 3645 encodes an assumption about Bambu slicer behavior that may be wrong. If per-plate `slice_info.config` files represent independent contributions (sum-correct), max-wins will systematically under-report consumption on any multi-plate print where the same slot is used across plates. This must be validated empirically before merging.

2. **[MEDIUM] Malformed sibling XML aborts entire cache read.** A ParseError in any sibling plate file triggers `3MF_CACHE_ERROR` and returns False, discarding the primary plate data. The sibling parse should be guarded per-file so one bad plate does not lose all data.

3. **[MEDIUM] FTPS success after disk-restore recheck miss leaves `_threemf_from_disk_restore = True`.** Site 6 (line 1956) does not reset the flag. A subsequent scheduling of `_fetch_3mf_background` would re-enter the recheck branch unnecessarily. The fix is one line: set `_threemf_from_disk_restore = False` at site 6.

4. **[LOW] Disk-restore recheck can overwrite better data with worse data.** If the recheck fires and the cache now has fewer plates (e.g., cache was partially cleared), `_threemf_data` is silently overwritten with inferior data. No guard or log exists for this regression.

5. **[LOW] Integration test gap for startup-rehydrate → recheck full path.** The primary use case (blip after rehydration triggers recheck) is not tested end-to-end.

**Recommendation:** Proceed with changes, but with the specific fixes identified above (items 2 and 3 are code-level fixes, item 1 requires Jerry's empirical confirmation, items 4 and 5 are test additions).

## Section 2: Principal Engineer Recommendation

### 1. Final Design
