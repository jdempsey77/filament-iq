### 0.2 `_collect_print_inputs` — how active slot is determined

**File: `apps/filament_iq/ams_print_usage_sync.py`**

Active slot tracking uses two layers:

**Layer 1 — `sensor.{prefix}_active_tray` listener** (lines 864–911):
`_resolve_active_tray_slot()` reads the `ams_index` and `tray_index`
attributes of the active tray entity (line 866–867), then looks up
`self._ams_tray_to_slot.get((int(ams_idx), int(tray_idx)))` (line 870).
`self._ams_tray_to_slot` is populated exclusively from `build_slot_mappings`
output (lines 170–175).

When the external spool is active, ha-bambulab will report
`ams_index=255, tray_index=0` on the active tray entity. The lookup
`self._ams_tray_to_slot.get((255, 0))` will return `None` (line 870), causing
`_resolve_active_tray_slot()` to return `None`. The `_on_active_tray_change`
handler (line 893–898) treats `slot is None` identically to "unavailable" —
it closes the active segment and clears `_current_active_slot`. The external
spool will never be added to `_trays_used`, so it will never enter
`_collect_print_inputs`.

**Layer 2 — `_collect_print_inputs`** (lines 432–541) iterates
`for slot in sorted(trays_used)`. If slot 8 is never added to `trays_used`,
it simply never appears. No explicit gap exists — slot 8 will be silently
invisible to consumption tracking.

**Conclusion:** Without `(255, 0) → 8` in `ams_tray_to_slot`, the external
spool is invisible to active-tray tracking and consumption. Slot 8 would need
an entry for the tray-active path to work. Alternatively, a separate listener
on `binary_sensor.{prefix}_externalspool_active` could directly manage
`_trays_used` — see Section 2 for recommendation.

### 0.3 `build_slot_mappings` extensibility

**File: `apps/filament_iq/base.py`, lines 25–57**

The function iterates `ams_units`, building entity IDs with the pattern:
```python
entity_id = f"sensor.{prefix}_ams_{ams_entity_idx}_tray_{tray_idx}"
```
It has no other entity naming assumptions. The entity name is constructed
entirely from `ams_entity_idx` (derived from `ams_index`) and the
slot-within-unit index `i`.

**The external spool sensor entity name is `sensor.{prefix}_externalspool_external_spool`** — it does not fit the `ams_{N}_tray_{N}` pattern. If you
naively add `{"type": "external", "ams_index": 255, "slots": [8]}` to
`ams_units`, `build_slot_mappings` will construct:
```python
entity_id = "sensor.{prefix}_ams_255_tray_1"  # WRONG
```
which does not exist in HA.

**There is no hardcoded pattern check** that would prevent a differently-named
entity from being mapped — the entity string is just stored in
`tray_entity_by_slot[8]`. The issue is only that the construction formula in
`build_slot_mappings` produces the wrong name for this unit type.

**Conclusion:** Slot 8 can be added to `tray_entity_by_slot` with a
non-standard entity name, but it cannot be done by adding a new `ams_units`
entry to the existing config structure without either (a) a special-cased
construction branch for `type=external`, or (b) a post-construction override
step in each app's `initialize()`. See Section 2 for the preferred approach.

### 0.5 Non-RFID reconcile path for external

**File: `apps/filament_iq/ams_rfid_reconcile.py`**

The non-RFID path is gated on `_is_all_zero_identity(tag_uid, tray_uuid)`
(line 1191) returning True. For external spool: `tag_uid = 0000000000000000`
and `tray_uuid` is presumably empty or all-zero. This satisfies `_is_all_zero_identity` (line 3231–3237) and gates into the non-RFID branch.

`_tray_meta(attrs, state_value)` at line 3192 reads:
```python
{
    "name":        attrs.get("name", state_value or ""),
    "type":        attrs.get("type", ""),
    "filament_id": attrs.get("filament_id", ""),
    "color":       attrs.get("color", ""),
    "color_hex":   color_candidates[0] if color_candidates else "",
}
```

The external spool sensor (`sensor.{prefix}_externalspool_external_spool`)
exposes the following attributes (per the brief):
- `type: PETG`
- `color: #F98C36FF` (RRGGBBAA, 8-char with alpha)
- `filament_id: GFG99`
- `tag_uid: 0000000000000000`

`_color_candidates()` (line 3688–3700) handles 8-char hex:
```python
if len(value) >= 8 and re.match(r"^[0-9a-f]{8}$", value):
    out.append(value[:6])  # strip alpha → "f98c36"
```
So `color_hex` will be correctly extracted as `f98c36`.

`_build_lot_sig(tray_meta)` (line 3290–3309):
```python
typ = "petg"
fid = "gfg99"
is_generic_filament_id("gfg99")  → True  (ends in "99")
# → returns "" immediately
```

**CRITICAL FINDING:** `GFG99` ends in `99`, making it a generic filament ID
(line 171–172: `endswith("99")`). `_build_lot_sig` returns `""` for generic
filament IDs (line 3302–3303). This means the external spool can never receive
a `lot_nr` enrollment from the standard non-RFID path. The lot_sig-based
matching (line 1391) would compute an empty sig and skip matching entirely.

`_build_lot_sig_for_lookup` has the same guard (line 3302 equivalent is lines
3318–3321) — it also requires non-generic `fid`.

The external spool would fall through to `NONRFID_NO_MATCH_CONFIDENT` or
similar, leaving it perpetually in a NEEDS_BIND state unless the user manually
assigns a spool_id via `FILAMENT_IQ_SLOT_ASSIGNED`.

**Note on `_build_lot_sig_for_lookup`:** The lookup variant (line 3311) does
allow generic filament IDs to pass through — but re-reading carefully, lines
3318–3321 show it has the same `if not typ or not fid: return ""` guard but
does NOT call `is_generic_filament_id`. So `_build_lot_sig_for_lookup` would
return `"petg|gfg99|f98c36"` even for a GFG99 spool. Only `_build_lot_sig`
(the write path) blocks generic IDs. This means lookup can succeed if a
Spoolman spool was previously enrolled with `lot_nr = "petg|gfg99|f98c36"` —
but the system would never auto-enroll such a lot_nr via `_build_lot_sig`.

**Conclusion for Section 0.5:** `type`, `filament_id`, and `color` attributes
exist on the external spool sensor and are readable by `_tray_meta`. Alpha
stripping works correctly in `_color_candidates`. The critical gap is that
`GFG99` is a generic filament ID, blocking automatic lot_sig enrollment. A
dedicated `_build_lot_sig_external` that bypasses the generic guard (but still
requires `type` and `color`) is needed, or the non-RFID reconcile path must be
explicitly configured for manual-assign-only on slot 8.

This is the only code path that uses `_ams_tray_to_slot` for runtime routing.
It reads `ams_index` and `tray_index` from the `sensor.{prefix}_active_tray`

it, the external spool is invisible to print-sync tray tracking. The reconciler
is unaffected because it ignores `ams_tray_to_slot`.

## Section 1: Skeptic (R1) Review

### 1.1 `ams_units` vs separate config

The proposed `external_spool_enabled: true` flat flag is simpler, but it
conflates two distinct concerns: (a) whether external spool support is active,
and (b) which slot number to assign it. A flat flag requires hardcoding
`SLOT_EXTERNAL = 8` in the engine code.

The `ams_units` approach (`{"type": "external", "slots": [8]}`) is consistent
with how the system already handles heterogeneous AMS hardware. It would make
slot 8 visible to all iteration loops without any special-casing, and it would
allow different users to assign different slot numbers (e.g., a user with a
second AMS HT at slot 8 would assign external to slot 9).

**Objection:** `ams_units` + `external` type requires branching in
`build_slot_mappings` to handle a non-standard entity name. The flat flag
approach pushes that branch into each app's `initialize()` instead. Neither
is clearly cleaner; the `ams_units` approach is more consistent with existing
architecture.

---

### 1.2 Entity naming assumption

`sensor.{prefix}_externalspool_external_spool` is derived from the ha-bambulab
integration's entity ID convention for `ExternalSpool`. This appears to be the
correct entity name based on the ha-bambulab source, but it has not been
verified on Jerry's actual HA instance.

The proposal watches only the sensor entity. The `binary_sensor.{prefix}_externalspool_active` entity reports when the printer is actively printing
from external. Two options for active detection:
1. Watch `active` attribute on the sensor entity directly (`sensor.{prefix}_externalspool_external_spool` has an `active` attribute per the brief).
2. Watch the separate `binary_sensor`.

**Objection:** The brief states `active = active_ams_index == 255 and active_tray_index == 0`. This information is already on `sensor.{prefix}_active_tray` — reading `ams_index` attribute. Watching the binary sensor separately would be redundant and create a second trigger path that could fire out of sync with the active-tray entity. The `_on_active_tray_change` listener approach (via `(255, 0) → 8` in `ams_tray_to_slot`) is architecturally cleaner.

---

### 1.3 "Configured but wrong spool" problem

When a user swaps the external spool but forgets to update Bambu Studio, the
sensor continues to report the old `type/filament_id/color`. Filament IQ
would consume from the wrong Spoolman spool record.

This is functionally identical to the same problem on non-RFID AMS HT slots.
The non-RFID reconciler already treats this as a known limitation — it detects
swaps via `_nonrfid_tray_matches_bound_spool()` (line 1258) which compares
current tray attributes against the bound spool. If the user changes the
configured filament in Bambu Studio, the attribute changes will trigger a
state-change event and the reconciler will reclassify the slot.

**Objection:** The external spool problem is slightly worse than AMS HT because
external spool swaps are more frequent (a dedicated bypass spool for flexible
materials, supports, etc.) and there is no physical "empty" state trigger when
the spool runs out. The `remain=-1` constraint means the system cannot detect
depletion from the sensor; it relies entirely on 3MF data or user declaration.

This is not a blocker but should be documented in the v1.9.0 release notes as
a known limitation.

---

### 1.4 3MF routing for external

Bambu Studio assigns `tray_info_idx` values in `slice_info.config`. For a
standard multi-color print with AMS slots 0–3, the values are `"0"`, `"1"`,
`"2"`, `"3"`. For a multi-color print that also uses external spool, the
`tray_info_idx` value assigned to external is unknown without examining an
actual 3MF file from such a print.

The current `match_filaments_to_slots` in `threemf_parser.py` ignores
`tray_info_idx` entirely — it matches by color and material
(lines 527, 537–543, 550–554). `tray_info_idx` is parsed and stored (line 387)
but not used in matching logic.

**Objection (critical):** If `tray_info_idx` for external is `"254"` or some
Bambu-specific sentinel, the current approach of ignoring it is fine —
color/material matching will match filament entries to slot 8 purely on
attributes. However, if Bambu Studio assigns `tray_info_idx="0"` to external
(treating it as an unindexed filament), it could collide with the first AMS slot
(index 0). This must be verified on a real multi-color print that uses both AMS
and external before any implementation proceeds.

**This is the single most important open question** (see Section 3).

---

### 1.5 Consumption double-attribution risk

The v1.7.4 set-once print identity invariant is enforced in `_on_print_end`
and the job-key dedup logic — a given job_key is only processed once. The
`_trays_used` set is populated by `_on_active_tray_change` events during a
single print lifecycle.

If `binary_sensor.{prefix}_externalspool_active` fires on/off transiently
while a print switches between AMS and external, `_trays_used` might include
both AMS slot(s) and slot 8. This is actually correct behavior — both were
active. The 3MF data (if available) will have the correct per-filament
attribution, and `match_filaments_to_slots` will distribute consumption to
the correct slots.

**No double-attribution risk from the tray-active tracking path.** The identity
guard is already in `_seen_job_keys`, which prevents re-processing. Within a
single print, if both an AMS slot and external are genuinely active, they each
get their own `SlotInput` and the engine decides independently.

---

### 1.6 OSS vs instance-specific boundary

The brief proposes that `external_spool_enabled` in `apps.yaml` is OSS, but
the slot 8 dashboard card is instance-specific. This is an inconsistent
boundary. If the config flag is OSS, other users will want a dashboard card.
If the dashboard card is instance-specific, the flag should also be
instance-specific (or at minimum, the OSS package should ship a reference
card template in `dashboard/`).

**Objection:** The `dashboard/` directory in the OSS repo already contains
reference card templates (`dashboard/` at repo root). Slot 8 support should
ship a reference card there, even if instance deployment is manual. Calling it
"instance-specific only" while shipping the config key OSS creates a confusing
half-baked experience for community users.

**Recommendation:** Ship everything OSS. The dashboard card belongs in
`dashboard/` as a reference. Users who don't use external spool set
`external_spool_enabled: false` (default) and ignore the card.

---

### 1.7 `SLOT_EXTERNAL = 8` collision risk

The current maximum slot number is 7 (AMS HT at slot 7, `ams_index=130`). The
`_default_ams_units()` function uses slots 1–7. If a user were to add a fourth
AMS HT at `ams_index=131`, it would naturally become slots 8–N using the
existing `ams_units` config pattern — colliding with `SLOT_EXTERNAL = 8`.

The ha-bambulab integration supports up to 4 AMS units (AMS Pro with 4 slots
or AMS HT units). A P1S supports 1x AMS 2 Pro (4 slots) + 4x AMS HT (4 slots
= 1 each), giving a theoretical maximum of 8 physical slots (1–8) with
external requiring slot 9.

**Objection:** Slot 8 is not safe. The correct slot number for external
depends on how many AMS units the user has. The proposed constant `SLOT_EXTERNAL = 8` should not be a hardcoded constant — it should either be derived from
`max(ams_unit_slots) + 1`, or configured explicitly as
`external_spool_slot: 8` (default) with documentation that it must not overlap
with any `ams_units` slot.

---

### Summary: Objections requiring resolution before implementation

**MUST RESOLVE (blocker):**
1. **`tray_info_idx` for external in Bambu Studio 3MF files** — unknown value.
   If it collides with index 0, matching breaks for multi-color prints. Needs
   empirical verification.
2. **`GFG99` is generic** — `_build_lot_sig` returns `""` for generic filament
   IDs, blocking lot_nr enrollment. The non-RFID lot_sig path is dead for
   external. Design must explicitly accept manual-assign-only OR add a
   `_build_lot_sig_external` bypass for the generic guard.
3. **Slot number collision risk** — `SLOT_EXTERNAL = 8` must not be a fixed
   constant. Must be configurable or derived.

**SHOULD RESOLVE (design clarity):**
4. **Entity name verification** — confirm `sensor.{prefix}_externalspool_external_spool` is the correct HA entity ID on the actual instance before
   hardcoding it.
5. **`ams_units` vs flat flag** — the flat flag approach requires invasive
   special-casing in `build_slot_mappings`. The `ams_units` + `type=external`
   approach is architecturally cleaner but requires a new entity construction
   branch.
6. **OSS boundary** — dashboard card should be in `dashboard/` (OSS), not
   instance-only.

**Recommendation:** PROCEED WITH CHANGES — the design is sound in principle
but has three specific gaps that must be addressed before implementation.

## Section 2: Principal Engineer Recommendation

### 2.1 Final config design

Use the `ams_units` extension approach, not a flat `external_spool_enabled`
flag. Add support for `type: external` in `build_slot_mappings`:

```yaml
ams_units:
  - {type: ams_2_pro, ams_index: 0, slots: [1, 2, 3, 4]}
  - {type: ams_ht, ams_index: 128, slots: [5]}
  - {type: ams_ht, ams_index: 129, slots: [6]}
  - {type: ams_ht, ams_index: 130, slots: [7]}
  - {type: external, slots: [8]}   # opt-in, not in _default_ams_units
```

This keeps `_default_ams_units()` returning the existing 7-slot layout (no
breaking change for existing users), while making external an explicit opt-in
via `ams_units` override. Users who do not override `ams_units` never see slot 8.

The `type: external` entry has no `ams_index` field (or `ams_index: 255`).

### 2.2 `build_slot_mappings` changes

Add a branch for `type == "external"` in `build_slot_mappings`:

```python
if unit_type == "external":
    # External spool uses a flat entity (no ams_N_tray_N pattern)
    assert len(slots) == 1, "external type supports exactly one slot"
    slot = int(slots[0])
    entity_id = f"sensor.{prefix}_externalspool_external_spool"
    tray_entity_by_slot[slot] = entity_id
    ams_tray_to_slot[(255, 0)] = slot   # active_ams_index=255, active_tray_index=0
    canonical_location_by_slot[slot] = "External"
    continue  # skip the standard tray loop
```

This is a 6-line addition to `build_slot_mappings` with no changes to the loop
logic for non-external units. Both apps pick it up automatically because they
call `build_slot_mappings` and use the returned maps directly.

The `continue` avoids re-entering the per-tray loop, keeping the logic clean.

### 2.3 `ams_tray_to_slot` entry for external

Yes, `(255, 0) → 8` must be added (handled in 2.2 above). Without it:

- `_resolve_active_tray_slot()` in print sync returns `None` when external is
  active.
- `_on_active_tray_change` closes the active segment and clears
  `_current_active_slot` (lines 893–898), meaning external spool is never
  added to `_trays_used`.
- No `_trays_used` entry → no `SlotInput` for slot 8 → no consumption recorded.

The reconciler ignores `ams_tray_to_slot` (discards it with `_` at line 392),
so this change has no side effects on reconciliation.

### 2.4 Reconcile cycle inclusion

If slot 8 is in `tray_entity_by_slot` at startup:

- `_physical_ams_slots` (line 394) automatically includes it.
- The reconcile loop at line 975 iterates `self._tray_entity_by_slot.items()`
  — slot 8 is included.
- The helper entity listener loop (lines 454–457) will register a listener on
  `input_text.ams_slot_8_spool_id` — this helper **must exist in HA** before
  deploying.
- The presentation-state validation loop (lines 493–497) checks
  `input_text.ams_slot_8_presentation_state` — this helper must also exist.

No `_physical_ams_slots` changes needed. Zero hardcoding of slot 8 outside the
`ams_units` config block.

### 2.5 Non-RFID attribute reading — gap analysis

| Attribute | External sensor name | `_tray_meta` key | Status |
|-----------|---------------------|-----------------|--------|
| `type`    | `type`              | `type`          | Direct match |
| `filament_id` | `filament_id` | `filament_id`   | Direct match |
| `color`   | `color` (RRGGBBAA) | `color` → `_color_candidates` → `color_hex` | Works — `_color_candidates` strips alpha at line 3691–3692 |
| `tag_uid` | `tag_uid` (all-zero) | read as `raw_tag_uid_ht` | All-zero correctly triggers non-RFID path via `_is_all_zero_identity` |

**Gap:** `GFG99` is a generic filament ID (`endswith("99")` → True). `_build_lot_sig` returns `""`, blocking lot_nr enrollment. This means:

- Auto-matching via lot_nr index (`lotnr_to_spools.get(lot_sig)` at line 1391)
  returns nothing, because no lot_sig was ever enrolled.
- The slot falls to `NONRFID_NO_MATCH_CONFIDENT` or needs manual assign.

**Recommendation:** Do not add a special `_build_lot_sig_external` bypass.
Instead, accept that external spool is **always manual-assign-only** — the user
must fire `FILAMENT_IQ_SLOT_ASSIGNED` to bind slot 8 to a Spoolman spool. This
is consistent with the semantics: the user is deliberately loading a specific
spool onto the external feed. The reconciler's `_on_slot_assigned` handler
(line 768) already handles this path and will accept any slot number.

Document this in the v1.9.0 user-facing notes: "External spool identity must
be assigned manually. Drag the spool card to slot 8, or fire the SLOT_ASSIGNED
event."

### 2.6 New presentation states

`EXTERNAL_NOT_BOUND` and `EXTERNAL_BOUND` as separate state constants are not
necessary. The existing states map correctly:

- `NEEDS_BIND_NONRFID` — slot 8 tray loaded, no spool assigned (unbound_reason
  = `UNBOUND_NO_RFID_TAG_ALL_ZERO` since tag_uid is all-zero).
- `BOUND` (or `OK_FORCE_ACCEPTED`) — once user assigns a spool.
- `EMPTY` — when external spool is not loaded (tray state = "Empty" or sensor
  `active: False` with empty state).

The only ambiguity is distinguishing "external slot is configured but no spool
is loaded" from "AMS slot is empty". Both currently produce `EMPTY`. If the
dashboard card for slot 8 has a distinct visual label ("External"), the state
machine output is unambiguous to the user without new constants.

If a dedicated label is strongly desired for "external slot has spool but
unbound", add `NEEDS_BIND_EXTERNAL` as a new state with dispatch rule:

```python
# In classify_slot_presentation, before NEEDS_BIND_NONRFID check:
if unbound_reason == _UNBOUND_NO_RFID_TAG_ALL_ZERO and is_external_slot:
    return NEEDS_BIND_EXTERNAL
```

But this requires passing `is_external_slot` to `classify_slot_presentation`,
breaking its pure-dispatch signature. Simpler: let the dashboard card know it
is slot 8 and render the label accordingly. The presentation state vocabulary
does not need to be extended.

**Decision:** No new state constants needed. Reuse `NEEDS_BIND_NONRFID` and
`BOUND`. Update `SLOT_PRESENTATION_LABELS` only if a distinct user-facing
label for external unbound is desired (optional, not a blocker).
